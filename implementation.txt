CrossCamReid — Multi-feed inference refactor
=============================================

Goal
----
Move from "1 worker thread per session that processes cameras serially" to a
real multi-feed architecture:

  - Single YOLO model loaded once into VRAM (was: N copies, one per camera).
  - One batched detection pass per tick across all cameras (single GPU forward).
  - Per-camera BYTETracker instances we manage ourselves so tracker state is
    isolated per stream (ultralytics' built-in track() shares one tracker
    across an in-memory batch, which would cross-contaminate IDs).
  - Per-camera writer threads with bounded drop-oldest queues so ffmpeg
    backpressure cannot stall the inference loop.
  - Existing per-camera TIDStateManager + ReID processor logic unchanged.

Architecture (after)
--------------------

  RTSPCapture(thread per cam)  -->  latest frame per cam
                                          |
                                          v
        +----------------------------------------------------------+
        | BatchedInferenceWorker (1 thread per session)            |
        |  - gather frames in fixed [cam0, cam1, ...] order        |
        |  - last-good frame as placeholder for stale slots so the |
        |    batch index <-> camera mapping never shifts           |
        |  - pose.predict([f0, f1, ...]) once  -> Results[]        |
        |  - per-cam BYTETracker.update(boxes, img) -> tracks      |
        |  - existing remap_tids + process_master/process_slave    |
        |  - draw_overlay(frame, records, ...)                     |
        |  - writer[cam].enqueue(annotated)  (non-blocking)        |
        +----------------------------------------------------------+
                                          |
                                          v
                  FFmpegHLSWriter (writer thread per cam)
                  - drains queue (maxsize=2, drop-oldest)
                  - synchronous .write() -> ffmpeg subprocess

Why per-camera BYTETracker (we own it)
--------------------------------------
ultralytics on_predict_start (trackers/track.py:61) only creates one tracker
per batch slot when dataset.mode == "stream" (live RTSP source). For an
in-memory list source the loop breaks after the first iteration, so all
batch items share one tracker -> cross-camera ID leak. To get both shared-
model VRAM savings AND isolated trackers, we drive BYTETracker manually:
one instance per camera, fed result.boxes.cpu().numpy() per tick.

Touched files (running log — append as work happens)
-----------------------------------------------------

[1] production/src/crosscamreid/server/ffmpeg_hls.py
    - Added bounded queue + writer thread to FFmpegHLSWriter.
    - New API: enqueue(frame) -> non-blocking, drop-oldest on full.
      Existing write(frame) preserved (still the synchronous ffmpeg call,
      now invoked from the writer thread instead of the inference thread).
    - New counter: frames_dropped (visible via property).
    - close() now signals the writer thread, drains, then tears down
      the ffmpeg subprocess as before.

[2] production/src/crosscamreid/server/hls_session.py
    - Replaced per-camera YOLO dict with a single shared `YOLO` instance.
    - Removed _run_frame() helper; replaced with batched per-tick logic.
    - Per-camera state:
        * BYTETracker      (owned by us, instantiated from cfg.runtime.tracker)
        * TIDStateManager  (unchanged)
        * FFmpegHLSWriter  (now async; we call enqueue() not write())
        * last_good_frame  (placeholder so batch indices stay stable)
    - Inference loop:
        * gather frames with last-good fallback per camera
        * single pose.predict(batch_frames, ...) call per tick
        * for each (cam, result) pair:
            - skip if cam had no fresh frame this tick
            - tracker.update(boxes_np, orig_img, feats) -> tracks
            - extract bbox + raw_tid + keypoints by `idx` column
            - remap_tids -> stable ids
            - process_master / process_slave per detection
            - draw_overlay -> writer.enqueue()
    - status() now exposes frames_dropped per camera.

[3] production/src/crosscamreid/server/__init__.py
    - No change needed (HLSStreamSession import path unchanged).

Out of scope (future work, intentionally NOT done now)
-------------------------------------------------------
- True async pipeline with separate decode/inference/encode stages connected
  by queues and multiprocessing (would also bypass the GIL).
- CV-CUDA / GPU preprocessing for the ReID crop path.
- IO binding for ONNXReIDBackend (TensorRTReIDBackend already uses pinned
  memory but ONNX backend still passes NumPy through onnxruntime).
- Frame-skipping policy for detection (run YOLO every K-th frame, advance
  trackers by Kalman in between).
- Triton / batched ReID across detections from all cameras.

------------

  RTSP sources aren't available in this environment).


People-counting + occupancy WebSocket layer (additive)
======================================================

Goal
----
Expose the existing ReID pipeline as a real-time people-counting +
occupancy API over WebSocket, without changing any existing ReID logic.

Endpoint
--------
  ws://<host>:8002/ws/people_counting/{client_id}?token=<JWT>

Run
---
  uvicorn app_people_counting:app --host 0.0.0.0 --port 8002

Auth
----
  - JWT (HS256) with secret from env PC_JWT_SECRET.
  - If PC_JWT_SECRET is unset, any non-empty token is accepted (dev only,
    one-time warning logged).

New files
---------
  production/src/crosscamreid/counting/
    __init__.py
    entry_exit.py     -- per-camera new-id->entry, timeout->exit accounting
    occupancy.py      -- threshold + edge-triggered alerts
    dwell.py          -- visible-avg dwell with last-10-exits fallback
    kvs_resolver.py   -- kvs:// -> HLS via boto3 (lazy import)
    gpu_stats.py      -- pynvml -> torch -> 0,0 graceful fallback
    auth.py           -- JWT validator
    org_registry.py   -- per-org shared SIDStore + ReID backend pool

  production/src/crosscamreid/websocket/
    people_counting_runner.py    -- PeopleCountingSession + per-cam worker
    people_counting_handler.py   -- start_stream / stop_stream / stop_all

  production/app_people_counting.py  -- FastAPI app on port 8002

Architecture
------------
  WebSocket (one per client)
       |
       v
  PeopleCountingSession (max 5 cameras)
       |
       +-- _CameraWorker (thread per camera)
       |     - YOLO instance (per camera) + RTSPCapture + TIDStateManager
       |     - shares ReID backend + SIDStore via OrgResources
       |     - process_master(...) reused unchanged from processor.py
       |     - EntryExitTracker + OccupancyTracker (edge-triggered alert)
       |     - emits one JSON payload per processed frame
       |
       +-- OrgRegistry (process-wide singleton in app)
              -> per-org Qdrant collection
                 (collection name: <base>__org<id>) so cross-org isolation
                 is strict

Cross-org isolation
-------------------
  - Each org_id has its own SIDStore-backed Qdrant collection.
  - A single WebSocket connection cannot mix org_ids; the handler refuses
    a start_stream that changes org_id mid-session.

Performance controls (handled in runner)
----------------------------------------
  - "fps"        -> min interval between processed frames per camera.
  - "frame_skip" -> drop K frames per processed one (additional throttle).
  - same-frame dedupe (capture hasn't refreshed) avoids re-processing.

Annotated frame
---------------
  Returned as null (no S3 in this environment). Hook for upload-then-URL is
  in _build_payload -> frame_data.annotated_frame_url; wire to S3 when
  available.

Out of scope (intentionally not done)
-------------------------------------
  - Line-crossing / zone polygons (entry/exit is timeout-based, which the
    spec explicitly allows: "ID disappears after timeout -> total_exits++").
  - S3 annotated-frame uploader.
  - Postgres write of counting frames (the existing ReID DB table is for
    the ReID stream payload; the new payload is a different schema).

Verification
------------
- python -m py_compile on all new modules: OK.
- Pure-python smoke tests for EntryExitTracker / OccupancyTracker /
  DwellTracker passed (see chat log).
- Manual smoke run requires GPU + a real RTSP/KVS feed (left to operator).
